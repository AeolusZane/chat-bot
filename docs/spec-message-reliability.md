# Spec: 消息可靠性 / 防丢失

> 状态：待评估，暂不实现。先记录方案，后续按需推进。

## 背景

机器人在线被动监听微信消息并落盘。当前实现存在丢消息风险：

1. **网络断开期间**：网页版协议（`wechaty-puppet-wechat`）断线时微信推送的消息，重连后不会重发 —— 协议层限制，应用层无法补救。
2. **进程崩溃/重启瞬间**：`fs.appendFile` 是异步写，缓冲区中的数据在崩溃时可能丢失。

## 根因

- 网页版微信本身无"离线消息漫游"，被动监听模式天然丢断线消息。
- 异步落盘存在窗口期。

## 方案分级（按投入从小到大）

| 优先级 | 做法 | 解决什么 | 成本 |
|--------|------|---------|------|
| P0 | `appendFile` → `appendFileSync`（或 fsync） | 进程崩溃时不丢已收到的消息 | 极低 |
| P0 | 用 `msg.id` 做幂等去重 | 为重放/补拉打基础，避免重复 | 低 |
| P1 | 换 padlocal（iPad 协议）puppet | 断线期间的离线消息可补拉 | 中（付费 token） |
| P2 | 引入 Redis Stream 缓冲队列 | 接收/处理解耦，支持重放 | 高 |

## 通用可靠性模式（参考）

- **WAL / 落盘优先**：收到消息第一件事同步落盘，再处理。
- **消息队列缓冲**：接收端只入队（Kafka/Redis Stream/RabbitMQ），处理端消费，解耦。
- **ACK + offset**：消费成功才提交位点，崩溃后从上次位点续传（at-least-once）。
- **幂等去重**：每条消息唯一 ID，存储前查重，配合 at-least-once 防脏数据。
- **断线重连 + 增量拉取**：重连后按 `since=上次时间戳` 补拉，前提是数据源支持历史查询。

## 务实结论

个人微信记录工具，**P0 两项基本够用**（同步落盘 + msg.id 去重）。MQ 那套是高并发服务端方案，当前场景过度设计，不做。

真要补断线期间的消息，唯一根治路径是换 padlocal 协议 —— 单独评估。
